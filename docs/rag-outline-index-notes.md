# RAG 链路与后续大纲索引优化记录

更新时间：2026-05-09

## 当前先保留的链路

现在先不继续把“内容分布”拆成更细的结构化索引，当前链路按下面方式跑通：

1. 上传文档后，系统为每个文档建立页级索引。
   - PDF：抽取页面文本、生成页面图像、写入全文检索索引和向量索引。
   - Excel：按表格/行内容建立可检索文本。
2. 索引过程中生成文档画像。
   - 文档类型
   - 摘要
   - 内容分布，也就是“哪页到哪页主要讲什么”
   - 关键词
   - 别名
3. 用户提问时，先做 query understanding。
   - 生成规范化问题、关键词、别名、可能章节、检索范围等。
   - 对英文缩写类问题，避免随意脑补无依据的展开词。
4. 文档路由先保持宽召回。
   - 如果用户明确点了某个文件或问题里明确匹配文件名，则只查对应文档。
   - 如果是多文档会话，则先让所有已就绪文档参与召回，不在最前面过早筛掉文档。
5. 每个候选文档内部做混合召回。
   - 关键词命中
   - SQLite FTS 全文检索
   - BAAI/bge-m3 向量检索
   - 必要时结合内容分布和相邻页扩展上下文
6. 多文档场景下，再用文档画像和召回结果做相关性过滤。
   - 这里看的不是单一 README 或短摘要，而是文件名、文档类型、摘要、内容分布、关键词、别名、召回页等综合信息。
   - 如果过滤模型异常，采用 fail-open，避免误删可能相关的文档。
7. 最终回答模型看到的是原始页面证据。
   - PDF 场景下发送页面图像和页面文本上下文。
   - 内容分布只作为导航参考，不能替代原始页面证据。

## 当前用到的主要工具和模型

- OCR：PaddleOCR / PP-OCRv5，用于 PDF 页面文字识别。
- PDF 页面处理：PyMuPDF，用于页面渲染、页面图像、页面文本处理。
- 全文检索：SQLite FTS，用于关键词和全文匹配。
- 向量检索：BAAI/bge-m3，用于语义召回。
- 大模型：qwen3.5-35b-a3b，用于 query understanding、文档画像、内容分布生成、文档相关性过滤和最终回答。
- 存储：SQLite，本地保存文档、页面、向量、文档画像和会话信息。

## 后续可尝试的优化：结构化大纲索引

这个优化可以参考 `llm-wiki-trial` 的思路，但先不做。核心想法是把现在一整段长文本形式的“内容分布”，升级成可检索、可排序、可定位的结构化条目。

每个条目可以类似：

```json
{
  "document_id": 20,
  "title": "MRB 处理程序",
  "start_page": 21,
  "end_page": 31,
  "summary": "说明材料审查委员会的组成、启动条件、处置选项和关闭周期要求。",
  "keywords": ["MRB", "处置", "返工", "报废", "特采"],
  "source": "profile_llm"
}
```

未来检索时可以多一层“大纲召回”：

1. 用户问题先做 query understanding。
2. 用问题和关键词检索结构化大纲条目。
3. 命中条目后展开到对应页范围，并加上前后邻近页。
4. 再和现有的关键词、FTS、向量召回结果合并。
5. 最终仍然把原始页面图像和页面文本交给回答模型。

这样做的潜在收益：

- 文档路由更稳定，尤其是多文档场景。
- 长文档定位更准，不必把很长的内容分布整段塞进 prompt。
- 可以更清楚地解释“为什么召回这些页”。
- 前端可以展示更清晰的目录式内容分布。
- 后续可以支持用户直接点某个内容块进入相关页。

需要注意的风险：

- 大纲是模型生成的二级信息，不能直接当事实证据回答问题。
- 最终回答必须回到原始页面文本或页面图像。
- 如果大纲页码范围生成错了，会影响召回，因此需要校验和回退机制。
- 需要新增表结构、迁移旧数据，并处理重建索引时的版本一致性。

## 如果以后要做，建议实施顺序

1. 先新增 `document_outline_entries` 表，保存标题、页码范围、摘要、关键词和 embedding。
2. 先从当前 `chapter_summary` 解析出条目，作为兼容旧数据的过渡方案。
3. 再改画像生成 prompt，让 LLM 直接输出 JSON 格式的大纲条目。
4. 为大纲条目建立 FTS 和向量索引。
5. 在检索日志里新增 `outline_pages` 或 `outline_entries` 字段，方便调试。
6. 前端默认展示简洁内容分布，点击后展开完整大纲。
7. 验证多文档问题、英文缩写问题、长 PDF 问题和无关文档过滤问题。

## 现在暂不做的原因

当前模型能处理 30 页左右的页面图像和文本，上下文窗口也足够大。对现在这个项目来说，直接把召回页面交给最终回答模型更简单，也避免为了二次筛选多花一遍模型 token。

所以当前策略是：

- 保留页级关键词、全文和向量召回。
- 保留内容分布作为文档导航和相关性过滤参考。
- 暂不把内容分布升级为独立的结构化 RAG 索引。
- 等后面遇到长文档定位不稳、多文档误召回明显、或者前端需要目录跳转时，再做这个优化。
